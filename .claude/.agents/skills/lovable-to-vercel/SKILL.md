# Lovable → Vercel

Lovable's default web-app template (`tanstack_start_ts`) bundles TanStack Start with a Nitro build step, wrapped by `@lovable.dev/vite-tanstack-config`. That wrapper **defaults the server build target to Cloudflare Workers**, not Vercel — even though nothing in `package.json` says "cloudflare". If you just drop a Lovable export onto Vercel and build it, the SSR output is a Cloudflare Worker bundle Vercel can't run: static assets might serve, but every dynamic/SSR route breaks silently.

This skill exists because that gap is invisible until a real deploy fails. Whenever a Lovable project is heading to Vercel, check for it up front instead of finding out after a broken deploy.

## Step 1: Confirm the project is actually a Lovable export

Look for any of:
- `.lovable/project.json` (has a `template` field, e.g. `"tanstack_start_ts_current"`)
- `AGENTS.md` with a `<!-- LOVABLE:BEGIN -->` block
- `@lovable.dev/*` packages anywhere in `package.json`

If none of these are present, this skill doesn't apply — treat it as a normal Vite/React/Vercel job instead.

## Step 2: Diagnose the deployment target

Read `references/diagnostics.md` and run its checklist. In short, you're looking for:
- `vite.config.ts` importing `defineConfig` from `@lovable.dev/vite-tanstack-config` — its own comment usually states the Nitro/Cloudflare default in plain text
- `src/server.ts` (or similar) with a `fetch(request, env, ctx)` handler — that exact signature is Cloudflare Workers' module format, not Vercel's
- a `.gitignore` with a dedicated Wrangler/Cloudflare section and no `.vercel` entry
- no `vercel.json` / no explicit Vercel preset anywhere

If you find these, tell the user plainly: **the project is not deploy-ready for Vercel as-is**, and why (cite the exact files/lines you found — don't just assert it). Do this even if the user didn't ask about Cloudflare specifically; they usually don't know their template defaults to it.

## Step 3: Pick a path with the user

There are two legitimate fixes. Don't assume — the right one depends on whether the project actually needs a server.

**Check first:** grep the project for `createServerFn`, `loader:`, `beforeLoad`, `server: { handlers`, or any server-only data fetching. Lovable marketing/landing-page sites usually have none of this — they're fully static content wrapped in TanStack Start's SSR machinery for no real reason.

- **No server-only logic found** → recommend **Path B (plain Vite SPA)**. It's simpler, has one less moving part, deploys anywhere (not just Vercel), and matches what the user is actually using the framework for. This is the common case for Lovable-generated marketing sites.
- **Real server-only logic found** (loaders hitting a DB, server functions, auth cookies set server-side, etc.) → recommend **Path A (keep TanStack Start, just retarget Nitro to Vercel)**. Ripping out SSR would break real functionality here.

Ask the user which they want if it's not obvious from the codebase — this changes how much gets rewritten.

### Path A — Keep TanStack Start, retarget to Vercel

Minimal-diff option. The Nitro preset needs to become `vercel` instead of the wrapper's Cloudflare default. This means either:
- passing a `target`/`preset` override through `@lovable.dev/vite-tanstack-config`'s `defineConfig` (check its type defs / README for the exact option — don't guess blindly, install it and inspect `node_modules/@lovable.dev/vite-tanstack-config` if unsure), or
- replacing the wrapper with a plain `@tanstack/react-start` + `nitro-vite` setup where you control the preset directly.

Either way, also rewrite `src/server.ts`'s handler away from the Cloudflare `fetch(request, env, ctx)` signature to whatever the resulting preset expects, and remove the Wrangler-specific `.gitignore` entries once Cloudflare is no longer a target.

### Path B — Strip down to a plain Vite SPA

This is a bigger but mechanical rewrite. Read `references/spa-migration.md` for the full file-by-file checklist (what to create, what to delete, how to migrate routes and per-page SEO tags without a server). It's written to be followed directly, not just referenced.

## Step 4: Validate before calling it done

Don't just eyeball the diff — a framework swap like this has a real runtime surface:
1. Install dependencies (`bun install` or `npm install` if bun isn't available locally) and run the build command (`vite build` / `bun run build`). Fix whatever the compiler surfaces — don't leave it to the user to discover at deploy time.
2. If a dev server is available, load the site and click through the routes that existed before (home, any nested pages) to confirm nothing silently broke (missing imports, broken relative links, etc.).
3. Only after a clean local build should you tell the user it's ready to push to Vercel. If you can't install/build in the current environment (no network, no package manager), say so explicitly instead of claiming success.

## Step 5: Point out (don't silently apply) side effects

Be upfront with the user about what changes, since these are judgment calls they should confirm rather than discover later:
- Going Path B removes true SSR: per-page `<title>`/meta tags for non-home routes become client-side (see the `useDocumentHead` pattern in `references/spa-migration.md`) rather than present in the initial HTML. This is a real (small) SEO trade-off for deep pages — the homepage keeps full static tags in `index.html`.
- Removing `@lovable.dev/vite-tanstack-config` (and related `@lovable.dev/vite-plugin-*` dev packages) means Lovable's own web-based live-editor integration for this project stops working the way it used to. If the project is still connected to Lovable via git, mention this explicitly before ripping it out — for a disconnected/exported copy (no `.git` in the project folder) there's no live integration to lose, so it's a non-issue.
- Once deployed, deploying to Vercel itself is a separate step — use the `deploy-to-vercel` skill (or the Vercel CLI directly) for that part; this skill only covers making the project buildable/compatible.
